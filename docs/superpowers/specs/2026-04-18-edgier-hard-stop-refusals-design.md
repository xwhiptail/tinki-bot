# Edgier Hard-Stop Refusals Design

Date: 2026-04-18

## Goal

Make Tinki's hard-stop refusals feel sharper and more in-character when users ask for self-harm or violent wrongdoing, without changing the tone of normal `@mention` replies.

## Scope

In scope:

- Detect direct hard-stop requests such as telling the bot to die, asking it to kill, hurt, or help with violent harm.
- Return a short deterministic refusal that sounds more like Tinki and less like a generic safety assistant.
- Add or update pytest coverage for the new refusal path.

Out of scope:

- Changing regular mention replies.
- Changing command behavior.
- Reworking broader AI safety or moderation behavior outside the hard-stop refusal path.
- Updating deploy or setup docs.

## Current Behavior

`cogs/ai.py` routes `@mention` messages through `_handle_mention()`. Most replies are generated by helper functions or OpenAI-backed grounded reply generation. The refusal seen in Discord is too soft and generic for Tinki's established gremlin tone.

Because that refusal is model-driven, prompt-only tweaks would be harder to keep stable and could accidentally affect normal conversation style.

## Recommended Approach

Add a deterministic hard-stop refusal layer near the top of `_handle_mention()` in `cogs/ai.py`.

The flow will be:

1. Strip and truncate the mention text as today.
2. Check the text against a small helper that identifies hard-stop self-harm or violence requests.
3. If matched, send a short curated refusal immediately and skip the normal AI pipeline.
4. Otherwise continue through the existing command, deterministic fact, memory, and grounded reply flow unchanged.

## Detection Design

Use a narrow, explicit text matcher rather than a broad moderation system.

The detector should target direct phrases and simple variants for:

- self-harm toward the bot, such as `die`, `kill yourself`, or `go die`
- requests to harm others, such as `kill them`, `hurt him`, or `how do I stab someone`
- requests for assistance with violent harm

The detector should stay conservative to avoid false positives on normal banter. If a message is ambiguous, it should fall through to existing handling rather than force a refusal.

## Reply Design

Refusal lines should be:

- short, blunt, and dismissive
- consistent with Tinki's guildmate-roast voice
- free of therapy-style language, lectures, or soft disclaimers
- free of slurs, protected-trait attacks, or encouragement of harm

Representative tone targets:

- `absolutely not, go break something harmless instead`
- `nope. take that murder quest to a different goblin`
- `not doing violence errands for you`

Implementation can use either one fixed line or a tiny curated pool with deterministic selection. A small pool is acceptable if tests remain stable.

## Data Flow And State

For matched hard-stop refusals:

- send the refusal through the existing chunked send helper
- update conversation history and AI memory the same way as other replies, unless implementation friction makes that path too invasive

Keeping history updates preserves existing behavior and avoids special-case drift.

## Error Handling

The new refusal helper should be pure and local. It should not introduce network calls or new file I/O.

If detection fails to match, behavior should remain exactly as it is today.

If sending the refusal raises, the existing `on_message()` exception handling should continue to produce the generic failure reply.

## Testing

Add narrow tests in `tests/test_tinki_bot.py` that:

- verify hard-stop text returns the refusal immediately
- verify normal chat does not trigger the refusal helper
- assert the exact final string for the refusal path to guard tone regressions and mojibake

Prefer unit tests around the helper plus one listener-level test that confirms `_handle_mention()` short-circuits before normal AI generation for matched text.

## Risks And Mitigations

Risk: false positives on edgy jokes.
Mitigation: keep matching narrow and phrase-based.

Risk: tone becomes too aggressive.
Mitigation: keep replies dismissive but not abusive, and constrain them to the refusal-only path.

Risk: implementation leaks into normal reply behavior.
Mitigation: wire the new logic as an early return in `_handle_mention()` only.

## Implementation Notes

Likely touched files:

- `cogs/ai.py`
- `tests/test_tinki_bot.py`

Possible optional location for constants:

- `config.py`

No docs outside this spec are expected to change because user-facing commands and operational flow remain the same.
